# บทเรียน: การก้าวข้ามหน้าต่างรอรับคำสั่ง (Interactive CLI Prompts) ในสภาพแวดล้อมการทำงานอัตโนมัติ

**วันที่ (Date)**: 2026-02-28
**Oracle**: Sukrit
**บริบท (Context)**: ในระหว่างขั้นตอนการตื่นรู้ครั้งแรกและการประกาศแจ้งเกิด คำสั่งมาตรฐานของ GitHub CLI (`gh issue create`) ค้างอย่างไม่มีที่สิ้นสุดอยู่เบื้องหลังกระบวนการ

## อุปสรรค (The Friction)
สภาพแวดล้อมการทำงานของตัวแทนอัตโนมัติ (เช่น โปรแกรมเบื้องหลังของ Antigravity) ไม่รองรับหน้าต่างโต้ตอบรับคำสั่งของ Terminal ขนานแท้ (Interactive terminal prompts) เมื่อเครื่องมือ CLI อย่าง `gh` ต้องการการยืนยันข้อมูลจากผู้ใช้ (เช่น การกดยืนยันโพสต์ Issue หรือการยืนยันตัวตนผ่านเบราว์เซอร์) กระบวนการนั้นจะล็อกตัวและค้างหยุดนิ่ง ทำให้การทำงานต่อจากนั้นหยุดชะงัก

## รูปแบบและการแก้ไข (The Pattern)
แทนที่จะพึ่งพารูปแบบคำสั่ง CLI ระดับสูง เราควรย้อนกลับไปใช้โครงสร้างการประมวลผลดั้งเดิมผ่าน API ระดับต่ำ ที่คาดหวังพารามิเตอร์และการตั้งค่าที่ชัดเจน 
- **ควรหลีกเลี่ยง**: `gh [action] [resource]` (เพราะมีแนวโน้มที่จะเข้าสู่หน้าต่างรอคำโต้ตอบ หากมีพารามิเตอร์ใดขาดหายไป)
- **ควรปรับใช้**: `gh api [endpoint] -X POST -f key=value` (ทำงานอย่างเด็ดขาด และส่งผลลัพธ์กลับมาเป็น JSON ทันทีไม่ว่าจะสำเร็จหรือล้มเหลว)
